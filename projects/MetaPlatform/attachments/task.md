## 0. Task

# Task: устранить деградацию времени reopen MetaGen params document после ввода данных в строках за пределами bootstrap, не ломая уже принятую auto-style семантику

## Контекст

Предыдущая задача по стабилизации и локальному рефакторингу `params table auto-style` считать выполненной на приемлемом уровне.

Новая задача отдельная и не является продолжением “рефакторинга ради чистоты”.

Сейчас проявляется конкретный баг производительности:
- создать новый MetaGen document;
- добавить строк больше bootstrap 100;
- ввести значение в строку примерно 105;
- закрыть документ;
- открыть документ снова;
- reopen происходит очень долго, вплоть до десятков секунд по пользовательскому наблюдению.

Нужно считать рабочей гипотезой, что проблема находится в reopen/init lifecycle `params` sheet, а не в общей платформенной логике открытия проекта.

По текущему коду видно вероятный проблемный путь:
- `params` sheet при reopen поднимается уже с runtime size, зависящим от persisted payload / actual shape;
- далее auto-style контур выполняет инициализационный refresh;
- этот refresh повторно читает диапазон строк из sheet;
- далее по строкам снова вычисляет категорию и применяет row-wide visual style/border;
- при строках за пределами bootstrap reopen-time path становится слишком дорогим.

Текущая итерация не про изменение пользовательской семантики строки `key/value/block`, а про устранение reopen-time performance bottleneck в пределах `params` sheet / auto-style lifecycle.

## Цель текущей итерации

Нужно:
1. Точно локализовать причину долгого reopen.
2. Отделить неизбежную работу при hydrate/open от случайно тяжёлой reopen-time переработки.
3. Внести минимально достаточное исправление.
4. Сохранить текущую рабочую auto-style семантику.
5. Не расширить задачу в общий platform refactor или новый auto-style engine.

Главный вопрос этой итерации:

**почему reopen params document после данных в строках > 100 проходит слишком долго, и как локально убрать эту стоимость без потери корректности поведения?**

## Главный технический фокус

Основной объект внимания — reopen/init path `MetaGen params` после hydrate persisted sheet data.

Особенно внимательно проверить следующие зоны.

### 1. Reopen pipeline `params` sheet

Нужно определить:
- где создаётся `params` sheet runtime;
- как выбираются runtime rows/cols при reopen;
- где hydrate переходит в visual initialization;
- где именно auto-style стартует при reopen;
- проходит ли reopen через путь, рассчитанный в основном на mutation handling, а не на open/init.

Ключевой вопрос:

**не используется ли для reopen слишком тяжёлый общий refresh path, который нормален для mutation flow, но слишком дорог для open flow?**

### 2. Повторное чтение и повторное применение style

Нужно проверить:
- читает ли reopen снова большой диапазон строк из sheet без реальной необходимости;
- пересчитывает ли категории строк повторно для всего диапазона;
- применяется ли `row-wide style` повторно к строкам, которые уже находятся в корректном визуальном состоянии;
- нет ли повторной тяжёлой работы по `setBackgroundColor / setFontWeight / setHorizontalAlignment / setVerticalAlignment / setBorder` на reopen.

Ключевой вопрос:

**не тратится ли reopen-time стоимость в основном на повторный heavy restyle уже готового диапазона?**

### 3. Разделение classify/state restore и expensive visual refresh

Нужно оценить:
- можно ли восстановить нужное runtime state легче, чем через полный/почти полный visual refresh;
- можно ли развести “понять категории строк” и “заново перекрасить диапазон”;
- можно ли сделать более узкий init path;
- можно ли сохранить safe fallback, но не выполнять дорогую работу всегда.

Ключевой вопрос:

**можно ли оставить reopen корректным, но сделать его заметно легче за счёт более узкого init path?**

### 4. Bootstrap boundary vs actual payload shape

Нужно проверить:
- как именно reopen ведёт себя при данных за пределами bootstrap 100;
- не становится ли строка 105+ триггером для тяжёлого full/near-full refresh;
- не завязан ли reopen path на фактический last-used row так, что инициализация деградирует непропорционально;
- не маскируется ли проблема под “большая таблица”, хотя реальная причина — лишний reopen-time visual recompute.

Ключевой вопрос:

**является ли деградация reopen следствием правильного роста runtime shape или следствием неправильной формы инициализации после этого роста?**

## Что именно требуется определить

Reviewer должен дать технически точный ответ на следующие вопросы:

1. Где находится узкое место reopen-time path.
2. Какая часть стоимости неизбежна, а какая случайна.
3. Выполняется ли при reopen лишний heavy auto-style refresh.
4. Можно ли локально уменьшить стоимость reopen, не меняя принятую семантику.
5. Нужен ли отдельный init/reopen path или достаточно сузить текущий refresh path.
6. Какой минимальный набор изменений действительно исправляет баг.
7. Как доказать, что баг устранён и поведение не деградировало.

## Пространство выбора reviewer

Reviewer не должен быть механически принуждён к одному конкретному виду решения.

Допустимы три исхода:

### Вариант A — root cause подтверждён, достаточно узкого performance fix

Тогда reviewer:
- локализует узкое место;
- даёт Codex минимальный implementation pass;
- ограничивает изменения reopen/init/auto-style lifecycle зоной;
- не допускает архитектурного разрастания.

### Вариант B — нужен небольшой рефакторинг reopen/init path

Тогда reviewer:
- разрешает локально перестроить reopen flow;
- сохраняет текущую auto-style семантику;
- не трогает несвязанные модули;
- не превращает задачу в переписывание `params table auto-style`.

### Вариант C — гипотеза о root cause оказывается неточной

Тогда reviewer:
- должен доказательно показать реальную причину;
- скорректировать план для Codex;
- всё равно удержать задачу в пределах `params` reopen / auto-style lifecycle;
- не уводить задачу в общий project-open refactor без очень сильного основания.

## Что следует считать хорошим результатом

Хороший результат этой итерации — это один из следующих исходов:

1. Локально устранён reopen-time bottleneck.
2. Reopen документа с данными в строках > 100 перестаёт деградировать катастрофически.
3. Текущая строковая auto-style семантика сохраняется.
4. Изменения остаются минимальными и понятными.
5. Reviewer может технически внятно объяснить:
   - где был bottleneck;
   - что именно изменено;
   - почему теперь reopen дешевле;
   - почему correctness не ухудшилась.

## Что именно нужно сохранить

Независимо от масштаба изменений должны сохраниться следующие свойства:

1. Таблица параметров остаётся строково-семантической.
2. Нельзя возвращаться к логике покраски одной отдельной ячейки там, где семантика строковая.
3. Уже подтверждённое UI-поведение должно сохраниться.
4. Existing ordinary edit / range edit / cut-like / move-like / structural row / structural column сценарии не должны деградировать.
5. Если где-то нужен safe fallback refresh, его допустимо оставить.
6. Нельзя ради ускорения reopen сломать correctness после mutation-сценариев.
7. Нельзя ухудшить поведение ради “более красивой” архитектуры.

## Что именно нужно улучшать, если улучшение оправдано

### 1. Reopen/init path

Если reopen действительно заходит в слишком тяжёлый общий refresh path, нужно:
- либо сузить этот путь;
- либо выделить более лёгкий init path;
- либо убрать лишний reopen-time restyle;
- но не менять принятую семантику таблицы.

### 2. Heavy visual restyle cost

Если узкое место находится в повторном применении row-wide style/border, нужно:
- понять, когда эта работа действительно необходима;
- убрать избыточные повторные visual operations;
- не выкинуть те части, которые реально нужны для correctness.

### 3. Runtime state restoration

Если reopen может быть ускорен через более дешёвое восстановление row state / categories, нужно:
- предпочесть это полному тяжёлому reopen refresh;
- не строить новый engine;
- не усложнять pipeline сильнее, чем нужно для исправления бага.

### 4. Доказуемость результата

Если нужно, допустимо:
- добавить targeted regression test;
- усилить проверки reopen behavior;
- зафиксировать, что reopen after rows beyond bootstrap больше не уходит в тяжёлый ненужный путь;
- при необходимости проверять не wall-clock, а наблюдаемое поведение/количество вызовов/объём heavy operations.

## Что НЕ нужно делать

1. Не переписывать весь `params table auto-style`.
2. Не менять строковую семантику таблицы.
3. Не строить новый global full-table engine.
4. Не втягивать в задачу `data table`, `MetaLab`, `MetaView` и несвязанные модули.
5. Не трогать IPC/preload/renderer/main boundary.
6. Не выносить задачу в общий platform open/save refactor без очень сильного доказательства.
7. Не делать performance-оптимизацию “на глаз” без локализации root cause.
8. Не делать рефакторинг ради рефакторинга.
9. Не спорить заново с уже принятыми архитектурными ограничениями и исключениями.
10. Не ухудшать correctness ради ускорения.

## На что обратить особое внимание при технической оценке

### 1. Где именно находится основная стоимость

Нужно не просто сказать “reopen медленный”, а локализовать:

## Требования к итоговой фиксации результата

В финальном summary обязательно нужно явно разделить:

- убедиться, что изменения в проекте не требуют обновления whitelist / project files logic;
- при необходимости скорректировать его только если это действительно требуется из-за состава проекта.

### `export_project_to_txt.py`

В этой задаче:
- **не запускать `export_project_to_txt.py`;**
- **не генерировать dump;**
- **не считать выполнение скрипта допустимой формой проверки.**

Проверка `export_project_to_txt.py` в этой задаче должна быть **только статической**:
- проверить, что файл не сломан;
- проверить, что текущие изменения не требуют обновления его ручного whitelist / project files logic;
- если обновление не требуется — явно так и написать;
- если обновление требуется — изменить только список/логику, но **не выполнять сам скрипт**.

Создание dump-файла в ходе этой задачи считается ошибкой выполнения задачи.

## Политика reviewer / question / dump

1. Reviewer не должен задавать вопрос пользователю, пока Codex ещё ни разу не выполнил ни одного реального implementation pass.

3. После первого реального implementation pass reviewer может задавать `question`, если это действительно полезно:
   - для ручной UI-проверки;
   - для критического решения пользователя;
   - для запроса дополнительного контекста;
   - для запроса dump.
4. Reviewer **может запросить dump**, если он ему реально нужен.
5. Но reviewer должен сначала попытаться разрулить ситуацию по:
   - текущему коду репозитория;
   - technical channel;
   - changed files;

7. Dump, если он нужен, предоставляет пользователь.
8. Reviewer не должен:
   - создавать dump;
   - править dump;
   - запускать dump-generating helper ради получения dump.
9. Перед **полным завершением задачи** reviewer **в любом случае обязан вернуть `status = "question"` и спросить пользователя, завершать ли работу или нет**.
