# CODE OVERVIEW / ARCHITECTURE EXCEPTIONS REGISTRY (MetaPlatform)

Этот документ нужен для будущих полных аудитов кода и для постановки задач Codex.

Цель:
- не возвращаться повторно к уже согласованным проектным допущениям;
- не трактовать повторно как “дефекты” то, что уже принято осознанно;
- отделять:
  - жёсткие архитектурные правила,
  - сознательно принятые исключения,
  - отложенный технический долг,
  - наблюдения, которые важно помнить, но не нужно автоматически превращать в bugfix.

---

## 0. HARD CONSTRAINTS / НЕ ПЕРЕСПОРИВАТЬ БЕЗ ОТДЕЛЬНОГО РЕШЕНИЯ

Ниже перечислено то, что **не нужно заново пытаться “исправлять” или “улучшать”** без отдельного прямого решения.

### 0.1. project.yaml не является полным source of truth для структуры проекта
Статус: СОЗНАТЕЛЬНО ПРИНЯТО.

Правило:
- структура платформы фиксированная;
- project.yaml не обязан определять архитектуру проекта целиком;
- список документов определяется сканированием папок.

Вывод:
- это проектное решение, а не дефект;
- не поднимать повторно как архитектурную претензию.

### 0.2. Имя проекта определяется именем project file
Статус: СОЗНАТЕЛЬНО ПРИНЯТО.

Правило:
- имя проекта = имя project file;
- переименование проекта допускается только через Save As.

Вывод:
- это целевая семантика, а не ошибка;
- не спорить с этим повторно как с недоработкой.

### 0.3. Placeholder-команды MetaLab / MetaView
Статус: СОЗНАТЕЛЬНО ПРИНЯТО.

Правило:
- модули пока архитектурно учитываются, но не реализованы полностью;
- наличие placeholder-команд допустимо на текущем этапе.

Вывод:
- не поднимать это повторно как “кривой код”;
- вернуться только при начале полноценной реализации этих модулей.

### 0.4. Inline rename / slow double click rename в дереве
Статус: СОЗНАТЕЛЬНО НЕ ИСПОЛЬЗУЕТСЯ.

Правило:
- переименование в дереве пока не планируется.

Вывод:
- само наличие отключенного поведения не считать проблемой;
- не поднимать это как обязательную доработку, пока не изменится UX-решение.

### 0.5. documentSnapshot через JSON.stringify(stableSortObject(...))
Статус: ПРИНЯТО КАК ДОПУСТИМОЕ РЕШЕНИЕ.

Правило:
- в этом проекте не ожидаются сложные вложенные спецтипы и нестабильные структуры;
- для текущей модели это допустимая реализация.

Вывод:
- не возвращаться к этому замечанию без появления новой фактической сложности формата.

### 0.6. Helper-скрипты используют ручной whitelist / PROJECT_FILES
Статус: СОЗНАТЕЛЬНО ПРИНЯТО.

Правило:
- оба helper-скрипта обязаны использовать жёстко заданный вручную перечень файлов;
- это обязательное требование проекта.

Вывод:
- нельзя предлагать автосканирование как “улучшение”, если нет отдельного прямого решения изменить подход;
- не трактовать ручной whitelist как баг.

### 0.7. export_project_to_txt.py экспортирует whitelist, а не “весь честный проект”
Статус: СОЗНАТЕЛЬНО ПРИНЯТО.

Правило:
- export_project_to_txt.py экспортирует whitelist;
- это обязательное требование проекта.

Вывод:
- не поднимать повторно как архитектурную претензию.

### 0.8. IPC-дублирование между config/platform-config.js и preload.cjs
Статус: СОЗНАТЕЛЬНО ПРИНЯТО НА ТЕКУЩЕМ ЭТАПЕ.

Причина:
- граница ESM / CJS / Electron preload / renderer-Vite технически чувствительна;
- попытки вынести IPC в единый shared contract уже приводили к регрессиям startup/preload.

Правило:
- `platform-config.js` хранит полный IPC contract для config/main-контура;
- `preload.cjs` хранит полный IPC contract для preload-контура;
- `tests/configWiring.integration.test.js` обязан проверять полное совпадение этих двух представлений.

Вывод:
- это не считается дефектом в текущей фазе проекта;
- **ВАЖНО: на данном этапе не нужно пытаться сделать единый shared contract**;
- не объединять preload/main и renderer IPC contract “ради дедупликации”.

### 0.9. generated/ не участвует в обычном project save
Статус: РЕШЕНИЕ ПРИНЯТО.

Правило:
- `generated/` = результаты генерации;
- обычный save проекта не очищает `generated/`.

Вывод:
- это уже не спорное поведение;
- не трактовать это как defect of save semantics.

### 0.10. Рекурсивное сканирование модульных папок
Статус: СОЗНАТЕЛЬНО ПРИНЯТО.

Пояснение:
- на текущем этапе проект в основном использует одноуровневую модель документов;
- при этом рекурсивный обход модульных папок оставлен сознательно как задел на будущее;
- это допускает возможное введение пользовательских подпапок и более сложной структуры дерева проекта без пересборки scan-layer.

Вывод:
- само наличие рекурсии не считать архитектурным дефектом;
- считать проблемой только если появятся реальные конфликты save/discovery semantics или UX-ошибки.

### 0.11. production/package-oriented замечания
Статус: ПОКА НЕПРИОРИТЕТНО.

Правило:
- до production/package ещё рано.

Вывод:
- не поднимать как значимый дефект, если пользователь не просит именно packaging/production review.

### 0.12. .gitignore / package-lock / воспроизводимость сборки
Статус: ПОКА ОТЛОЖЕНО.

Правило:
- тема не закрыта навсегда, но сейчас не является рабочим приоритетом.

Вывод:
- не включать в ключевые замечания к текущей кодовой базе без отдельного запроса.

## 1. ЯВНЫЕ АРХИТЕКТУРНЫЕ ПРАВИЛА (ЭТО УЖЕ НЕ СПОРНЫЕ ВОПРОСЫ)

Ниже перечислено то, что нужно считать **принятой архитектурой проекта**, а не “одной из гипотез”.

### 1.1. project-node / header
Статус: РЕШЕНИЕ ПРИНЯТО.

Правило:
- header панели проекта остается status-line;
- root-node в дереве существует отдельно как структурный корень;
- root-node имеет постоянное имя “Проект”.

Вывод:
- это больше не спорный вопрос.

### 1.2. GUID как единственная внутренняя identity
Статус: РЕШЕНИЕ ПРИНЯТО.

Правило:
- GUID = единственная внутренняя identity документа;
- `name` / `path` / `file name` — только UI/storage representation.

Вывод:
- это не обсуждать заново как гипотезу;
- считать базовым правилом проекта.

### 1.3. project.yaml не хранит список документов проекта
Статус: РЕШЕНИЕ ПРИНЯТО.

Правило:
- список документов определяется сканированием модульных папок.

Вывод:
- не поднимать повторно как “архитектурный дефект”.

## 2. ОСОЗНАННЫЙ ТЕХНИЧЕСКИЙ ДОЛГ (НЕ ИСПРАВЛЯТЬ ПО УМОЛЧАНИЮ, НО И НЕ СЧИТАТЬ ЗАКРЫТЫМ)

Ниже перечислено то, что **не нужно автоматически исправлять в каждом цикле**, но нужно помнить как реальный отложенный долг.

### 2.1. Неатомарное удаление документа
Статус: ПОКА ОСТАВЛЕНО КАК ЕСТЬ.

Как трактовать:
- не считать это “забытым”;
- не требовать исправления по умолчанию в каждом цикле;
- помнить, что это технический долг и потенциальный источник рассинхрона runtime/disk.

Когда вернуться:
- при усилении file lifecycle;
- при усилении transactional guarantees;
- при доработке delete flow и recovery semantics.

### 2.2. Возможный runtime/disk рассинхрон при rename документа
Статус: ПОКА ОСТАВЛЕНО КАК ЕСТЬ.

Как трактовать:
- не поднимать как уже согласованную срочную правку;
- считать отложенным техническим долгом.

Когда вернуться:
- при следующем цикле усиления rename/save-as/path remap semantics;
- если появятся реальные сбои;
- если усложнится модель persisted/unsaved documents.

### 2.3. Ошибочные документы тихо пропускаются при сканировании
Статус: ПОКА ОСТАВЛЕНО КАК ЕСТЬ.

Как трактовать:
- не считать это закрытым “идеальным” поведением;
- это временно допустимая политика.

Когда вернуться:
- когда потребуется более строгая диагностика поврежденных проектов;
- когда появится отдельная задача на UX ошибок загрузки/валидации проекта.

### 2.4. Production/package контур
Статус: ОТЛОЖЕНО ДО НУЖНОЙ ФАЗЫ ПРОЕКТА.

Когда вернуться:
- при подготовке к packaging;
- при release-сборке;
- при распространении вне dev-среды.

### 2.5. package-lock / reproducibility / dependency discipline
Статус: ОТЛОЖЕНО ДО СТАБИЛИЗАЦИИ КОНТУРА.

Когда вернуться:
- при стабилизации архитектурного контура;
- при подготовке более строгой build/release дисциплины.

## 3. ЧАСТИЧНО ПРИНЯТЫЕ НАБЛЮДЕНИЯ (НЕ BUGFIX ПО УМОЛЧАНИЮ, НО ВАЖНО ПОМНИТЬ)

### 3.1. Покрытие тестами неравномерное
Статус: НЕ СЧИТАТЬ ОТДЕЛЬНЫМ ДЕФЕКТОМ “САМО ПО СЕБЕ”.

Что уже зафиксировано:
- после любых изменений всегда прогоняются абсолютно все тесты;
- это обязательное правило проекта.

Как трактовать:
- вопрос не в частичном запуске, а в достаточности самого набора тестов;
- если в полном аудите всплывает архитектурный контур без покрытия, это можно отмечать;
- но не нужно заново объяснять правило полного прогона — оно уже зафиксировано.

### 3.2. Тесты больше покрывают happy-path и rollback, чем часть архитектурных инвариантов
Статус: НАБЛЮДЕНИЕ, А НЕ АВТОМАТИЧЕСКИЙ БЛОКЕР.

Как трактовать:
- можно поднимать, если это реально мешает задаче;
- но не нужно каждый раз заново спорить о самом принципе полного прогона — он уже обязателен;
- нужно смотреть именно на достаточность покрытия под конкретную задачу.

### 3.3. Полнота архитектурного покрытия тестами
Статус: ВЕРНУТЬСЯ ПОЗЖЕ, НО НЕ СЧИТАТЬ ЗАКРЫТЫМ НАВСЕГДА.

Когда вернуться:
- при следующих больших изменениях в core/runtime;
- при появлении новых системных контуров;
- при необходимости закрепить новые архитектурные инварианты отдельными тестами.

## 4. К ЧЕМУ НУЖНО ОБЯЗАТЕЛЬНО ВЕРНУТЬСЯ ПОТОМ

Это не исправляется прямо сейчас по умолчанию, но и не должно раствориться как “когда-нибудь потом”.

### 4.1. Устойчивость delete flow
Вернуться при:
- следующем цикле file lifecycle hardening;
- необходимости stronger transactional guarantees;
- появлении recovery/undo/repair semantics.

### 4.2. Устойчивость rename flow и consistency runtime/disk
Вернуться при:
- следующем цикле усиления rename/save-as/remap;
- расширении persisted/unsaved lifecycle;
- появлении новых типов документов/модулей.

### 4.3. Диагностика поврежденных документов при open/scan
Вернуться при:
- задаче на project validation / diagnostics UX;
- появлении требований явно показывать broken files, а не тихо пропускать их.

### 4.4. Production/package контур
Вернуться при:
- подготовке к packaging;
- release-сборке;
- распространении вне dev-среды.

### 4.5. package-lock / reproducibility / dependency discipline
Вернуться при:
- стабилизации архитектурного контура;
- подготовке более строгой build/release дисциплины.

### 4.6. Полнота архитектурного покрытия тестами
Вернуться при:
- следующих больших изменениях в core/runtime;
- появлении новых системных контуров;
- необходимости закрепить новые архитектурные инварианты отдельными тестами.

## 5. ИНСТРУКЦИЯ ДЛЯ БУДУЩИХ ПОЛНЫХ АУДИТОВ КОДА

Когда в следующий раз проводится полный аудит кода, **не нужно заново трактовать как замечания** следующее:

- project.yaml не полный source of truth;
- имя проекта = имя project file;
- placeholder MetaLab/MetaView;
- отключенный rename в дереве;
- ручной whitelist helper-скриптов;
- whitelist export;
- generated/ отдельно от обычного save;
- GUID как единственная identity;
- header + root-node как разделённые роли;
- documentSnapshot через stableSortObject + JSON stringify;
- не-приоритетность production/package замечаний прямо сейчас;
- IPC-дублирование между `config/platform-config.js` и `preload.cjs` как сознательно допустимое архитектурное исключение на текущем этапе;
- рекурсивное сканирование модульных папок как допустимый задел на будущее.

Но нужно помнить как отложенный технический долг:

- неатомарное удаление документа;
- возможный runtime/disk рассинхрон при rename;
- тихий пропуск ошибочных документов при scan/load;
- future work по packaging/release;
- future work по package-lock / reproducibility / dependency discipline;
- future work по полноте архитектурного тестового покрытия.

## 6. КАК ИСПОЛЬЗОВАТЬ ЭТОТ ДОКУМЕНТ В ЗАДАЧАХ ДЛЯ CODEX

Перед любым полным аудитом или крупным рефакторингом нужно сначала проверить этот документ и разделить все наблюдения на четыре класса:

1. **Hard constraints / не трогать без отдельного решения**
2. **Принятая архитектура / не спорить заново**
3. **Отложенный технический долг / помнить, но не чинить автоматически**
4. **Наблюдения / поднимать только если они реально мешают текущей задаче**

Особенно важно:
- не пытаться автоматически “дедуплицировать” IPC-контракт через shared config;
- не трактовать helper whitelist как дефект;
- не трактовать project.yaml как обязательный полный реестр документов;
- не тащить packaging/release замечания как основной приоритет до соответствующей фазы проекта.

- не тратить время на повторный спор по уже принятым решениям;
- не подменять реальные приоритеты абстрактным “улучшайзингом”;
- не ломать рабочие архитектурные исключения ради красивой дедупликации;
