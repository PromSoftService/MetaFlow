# ARCHITECTURE EXCEPTIONS REGISTRY — METAPLATFORM

Актуальная консолидированная версия для будущих полных аудитов, промежуточных проверок и постановки задач reviewer → codex.

Этот документ нужен для того, чтобы:
- не спорить повторно по уже принятым архитектурным допущениям;
- не поднимать как дефекты то, что принято сознательно;
- отделять hard constraints, принятые решения, исключения, техдолг и просто наблюдения.

Документ должен быть устойчив к дальнейшему перемещению файлов внутри репозитория.

---

## 0. HARD CONSTRAINTS — НЕ ПЕРЕСПОРИВАТЬ БЕЗ ОТДЕЛЬНОГО РЕШЕНИЯ

### 0.1. Проект остаётся модульной платформой, а не MetaGen-only приложением
Это не обсуждать заново в обычных рефакторинг-задачах.

### 0.2. Web-only курс зафиксирован
- Electron не возвращать как целевую runtime оболочку.
- Desktop-shell compatibility не считать обязательным направлением по умолчанию.
- Packaging desktop-приложения не поднимать как приоритет без отдельного прямого решения.

### 0.3. Проект = директория с фиксированными модульными зонами
- `project.yaml` обязателен.
- Список документов не обязан храниться в `project.yaml`.
- Модульные папки определяются архитектурой.

### 0.4. Имя проекта = project file / Save As
- Имя проекта определяется project file.
- Отдельную независимую name-identity не вводить.
- Переименование проекта как отдельный inline/project-title flow не считать обязательным.

### 0.5. GUID = единственная runtime identity документа
- Не возвращаться к path/name/file-name identity semantics.
- Name/path/file name — только presentation/storage attributes.

### 0.6. `generated/` не участвует в обычном project save
- `generated/` = generation outputs.
- Обычный save не должен очищать `generated/`.
- Это уже принятая семантика, не спорный пункт.

### 0.7. Helper scripts используют ручной whitelist
- `create_project_structure.py` и `export_project_to_txt.py` должны оставаться whitelist-based, если отдельным решением не изменён сам подход.
- Автосканирование не предлагать как “очевидное улучшение” по умолчанию.

### 0.8. Export helper экспортирует whitelist, а не “весь честный repo dump”
Это сознательное правило проекта.

### 0.9. Repo normalization идёт поэтапно
Не смешивать в одной задаче несколько больших шагов layout migration, если это можно сделать отдельными инкрементами.

## 1. ПРИНЯТАЯ АРХИТЕКТУРА — ЭТО УЖЕ НЕ СПОРНЫЕ ВОПРОСЫ

### 1.1. Целевой repo layout
Целевая структура репо:
- `ui/`
- `rtb/`
- `shared/`
- `tools/`
- `docs/`

Это не “одна из идей”, а текущий принятый курс нормализации репозитория.

### 1.2. Shared config layer допустим и нужен
Если literal/contract реально общий для нескольких слоёв и не ломает boundary, его допустимо и желательно держать в `shared/`.

### 1.3. UI и backend должны оставаться разделёнными контурами
- Frontend и runtime backend — разные контуры.
- Repo cleanup не должен смешивать backend move с раздачей built UI или другим соседним scope.

### 1.4. Helper scripts должны синхронно отражать фактический layout
При любом переносе каталогов/helper paths оба helper-скрипта должны обновляться вместе.

### 1.5. Два briefing-документа не являются обязательной частью repo layout
`01_project_doctrine.md` и `03_architecture_exceptions.md` допустимы как внешние документы для reviewer, но их не нужно считать обязательными repo artifacts и не нужно возвращать в helper whitelist/export.

---

## 2. СОЗНАТЕЛЬНО ПРИНЯТЫЕ ИСКЛЮЧЕНИЯ

### 2.1. Частично реализованные MetaLab / MetaView допустимы
Пока модуль архитектурно присутствует, placeholder-level реализация сама по себе не является дефектом.

### 2.2. Рекурсивное сканирование модульных папок допустимо
Даже если текущая модель в основном одноуровневая, рекурсивный scan-layer допускается как задел на будущее, пока он не создаёт реальных конфликтов save/discovery semantics.

### 2.3. Helper whitelist — это сознательное ограничение, а не баг
Ручное сопровождение `PROJECT_FILES` не трактовать как “грязный временный хак” само по себе.

### 2.4. Не всякий локальный literal является проблемой
Одноразовые локальные сообщения, не участвующие в wiring/contract semantics, допустимо оставлять локально.

### 2.5. Не нужно механически превращать repo cleanup в широкую архитектурную зачистку
Если задача про перенос каталогов и wiring, не нужно расширять её до unrelated core refactor без прямой причины.

---

## 3. ОТЛОЖЕННЫЙ ТЕХНИЧЕСКИЙ ДОЛГ — ПОМНИТЬ, НО НЕ ЧИНИТЬ ПО УМОЛЧАНИЮ

### 3.1. Неатомарные file lifecycle операции
Если где-то ещё остаются хрупкие rename/delete/save path flows, это помнить как техдолг, но не добавлять автоматически в любую соседнюю задачу.

### 3.2. Возможный runtime/disk рассинхрон вокруг rename/remap
Это реальный техдолг, но не универсальный обязательный scope каждой итерации.

### 3.3. UX/diagnostics для повреждённых документов
Тихий skip broken documents или недостаточно строгая диагностика — это будущий отдельный контур, а не автоматический bugfix в любой задаче.

### 3.4. Packaging/release/reproducibility дисциплина
- build reproducibility;
- package-lock policy;
- release packaging;
- production delivery discipline.

Это не нулевой приоритет, но и не то, что надо затаскивать в каждый текущий цикл.

### 3.5. Полнота архитектурного тестового покрытия
Это важный отложенный контур. Его надо усиливать по мере появления новых инвариантов, но не превращать в бесконечный общий scope.

---

## 4. ЧТО НЕ НУЖНО ПОВТОРНО ТРАКТОВАТЬ КАК ДЕФЕКТЫ

При полном аудите или постановке очередной задачи не поднимать повторно как замечания сами по себе:
- модульную природу платформы;
- web-only курс;
- layout-цель `ui / rtb / shared / tools / docs`;
- project-as-directory модель;
- неполный source-of-truth характер `project.yaml`;
- имя проекта через project file / Save As;
- GUID-only identity;
- `generated/` вне обычного save cleanup;
- ручной whitelist helper scripts;
- whitelist-based export;
- наличие placeholder-level модулей MetaLab/MetaView;
- допустимость рекурсивного scan-layer;
- то, что эти два briefing-doc не должны жить в repo как обязательные экспортируемые артефакты.

---

## 5. ЧТО НУЖНО ПРОВЕРЯТЬ В КАЖДОМ БОЛЬШОМ REVIEW

При любом полном аудите или крупном рефакторинг-цикле сначала проверять:
1. не нарушен ли текущий layout course;
2. не расползся ли scope за пределы задачи;
3. не появились ли новые magic strings / magic numbers вне config/constants слоя без причины;
4. синхронизированы ли helper scripts;
5. не вернулись ли в export/icons/demo/top-level tests/лишние docs;
6. соответствуют ли test/run commands фактическому layout проекта;
7. не появился ли compatibility-мусор после завершённого переноса.

---

## 6. КАК ИСПОЛЬЗОВАТЬ ЭТОТ ДОКУМЕНТ В ЗАДАЧАХ REVIEWER → CODEX

Перед новой задачей или аудитом разделять наблюдения на пять классов:
1. hard constraints;
2. принятая архитектура;
3. сознательно допустимые исключения;
4. отложенный технический долг;
5. реальные дефекты в scope текущей задачи.

Важно:
- не смешивать эти классы между собой;
- не раздувать текущую задачу техдолгом “раз уж уже полезли в этот файл”; 
- не предлагать дедупликацию или унификацию просто “ради красоты”, если она не даёт практического выигрыша в текущем scope.

---

## 7. КОРОТКАЯ СУТЬ ДОКУМЕНТА

Этот документ нужен не для абстрактной “идеальной архитектуры”, а чтобы:
- не спорить по второму кругу о решённых вещах;
- не путать принятые исключения с дефектами;
- удерживать reviewer в актуальном курсе проекта;
- не засорять задачи лишним scope;
- помнить реальный техдолг, но чинить его только тогда, когда он действительно входит в текущую задачу.
