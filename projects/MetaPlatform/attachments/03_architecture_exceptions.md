# ARCHITECTURE EXCEPTIONS REGISTRY — METAPLATFORM

Актуальная консолидированная версия для будущих аудитов, промежуточных проверок и постановки задач reviewer → Codex.

Этот документ нужен для того, чтобы:
- не спорить повторно по уже принятым архитектурным допущениям;
- не поднимать как дефекты то, что принято сознательно;
- отделять принятые исключения, техдолг и реальные дефекты текущего scope;
- не конфликтовать с фактической механикой цикла reviewer → Codex и артефактами итераций.

Документ должен быть устойчив к дальнейшему перемещению файлов внутри репозитория.

---

## 0. ЧТО ЯВНО НЕ СЧИТАТЬ ДЕФЕКТОМ

### 0.1. Частично реализованные MetaLab / MetaView допустимы
Пока модуль архитектурно присутствует, placeholder-level реализация сама по себе не является дефектом.

### 0.2. Рекурсивное сканирование модульных папок допустимо
Даже если текущая модель в основном одноуровневая, рекурсивный scan-layer допускается как задел на будущее, пока он не создаёт реальных конфликтов save/discovery semantics.

### 0.3. Helper whitelist — это сознательное ограничение, а не баг
Ручное сопровождение whitelist в `export_project_to_txt.py` не трактовать как “грязный временный хак” само по себе.

### 0.4. Export helper в root — не дефект
Нахождение `export_project_to_txt.py` в корне репозитория не трактовать как дефект само по себе.
Не предлагать перенос helper-а в `tools/` без отдельного решения и отдельного scope.

### 0.5. Не всякий локальный literal является проблемой
Одноразовые локальные сообщения и чисто технические локальные микролитералы, не участвующие в contract semantics и не создающие новый source of truth, допустимо оставлять локально.

### 0.6. Backend lock/process semantics — не implementation noise
Следующие состояния и ограничения не трактовать как случайный побочный эффект:
- backend project lock ownership;
- heartbeat/session lifecycle;
- lost lock handling;
- active process blocking;
- блокировка отдельных close/save/switch/open/export/import/delete transitions по backend conditions.

Это часть текущей operational semantics платформы.

### 0.7. Briefing docs не нужно править в каждой итерации
`01_project_doctrine.md` и `03_architecture_exceptions.md` не нужно менять автоматически в каждой задаче.
Их правка требуется только тогда, когда появилось реальное расхождение с актуальным состоянием проекта, с механикой reviewer → Codex или с принятыми постоянными правилами.

### 0.8. Наличие артефактов цикла не является частью repo diff
Артефакты итераций вроде `reviewer_response.json`, `codex_task.md`, `git_diff.patch`, `tests_summary.json`, `test_results.txt`, `codex_trace.jsonl` являются рабочими артефактами процесса review/execution.
Их наличие в orchestration artifacts не означает, что они должны попадать в исходный repo или в `export_project_to_txt.py`.

## 1. ЧТО НЕ НУЖНО ЧИНИТЬ БЕЗ ОТДЕЛЬНОГО SCOPE

### 1.1. Repo cleanup не превращать в широкую архитектурную зачистку
Если задача про конкретный UI flow, backend contract, helper/export, layout consistency или docs-sync, не нужно расширять её до unrelated core refactor без прямой причины.

### 1.2. Docs не расширять до общего docs-cleanup
Если README, runbook или другая документация меняются в пределах конкретной задачи, это не повод автоматически расширять scope до широкого общего пересмотра всей документации.

### 1.3. Layout cleanup не смешивать с соседними темами
Если задача про helper/export/layout consistency, не нужно автоматически добавлять:
- новый функционал;
- unrelated refactor;
- доп. UX-cleanup;
- произвольное упрощение архитектуры.

### 1.4. Orchestration mechanics не превращать в требования к app code
Reviewer может учитывать technical/model channel и артефакты итераций, но не должен переносить сущности процесса review/execution в код MetaPlatform без отдельного product scope.

---

## 2. ОТЛОЖЕННЫЙ ТЕХНИЧЕСКИЙ ДОЛГ — ПОМНИТЬ, НО НЕ ЧИНИТЬ ПО УМОЛЧАНИЮ

### 2.1. UX/diagnostics для повреждённых документов
Тихий skip broken documents или недостаточно строгая диагностика — это будущий отдельный контур, а не автоматический bugfix в любой задаче.

### 2.2. Packaging/release/reproducibility дисциплина
- build reproducibility;
- package-lock policy;
- release packaging;
- production delivery discipline.

Это не нулевой приоритет, но и не то, что надо затаскивать в каждый текущий цикл.

### 2.3. Полнота архитектурного тестового покрытия
Это важный отложенный контур. Его надо усиливать по мере появления новых инвариантов, но не превращать в бесконечный общий scope.

### 2.4. Единая русификация UI
Англоязычные пользовательские строки в UI нужно устранять в задачах, где затрагивается соответствующий экран/flow.
Не расширять любую задачу до тотальной русификации всего продукта без отдельного scope.

---

## 3. ЧТО НЕ НУЖНО ПОВТОРНО ПОДНИМАТЬ КАК ЗАМЕЧАНИЕ САМО ПО СЕБЕ

При полном аудите или постановке очередной задачи не поднимать повторно как замечания сами по себе:
- модульную природу платформы;
- web-only курс;
- GUID-only identity;
- ручной whitelist export helper;
- нахождение `export_project_to_txt.py` в root;
- наличие placeholder-level модулей MetaLab/MetaView;
- допустимость рекурсивного scan-layer;
- backend lock/process lifecycle semantics;
- то, что briefing docs не входят в repo export whitelist;
- то, что артефакты цикла reviewer → Codex существуют вне исходного repo;
- то, что reviewer принимает решение по diff/test artifacts, а Codex является исполнителем изменений и команд.

---

## 4. КАК REVIEWER ДОЛЖЕН КЛАССИФИЦИРОВАТЬ ПРОБЛЕМЫ

Перед новой задачей или аудитом разделять наблюдения на четыре класса:
1. принятые исключения;
2. отложенный технический долг;
3. реальные дефекты в scope текущей задачи;
4. ограничения среды запуска, подтверждённые конкретным stdout/stderr.

Важно:
- не смешивать эти классы между собой;
- не раздувать текущую задачу техдолгом “раз уж уже полезли в этот файл”;
- не предлагать дедупликацию или унификацию просто “ради красоты”, если она не даёт практического выигрыша в текущем scope;
- не называть падение assert/error mismatch ограничением среды;
- не называть отсутствие npm script, рассинхрон README/package.json/helper-script/test expectation ограничением среды без дополнительной проверки.

---

## 5. ОСОБЫЕ ОГРАНИЧЕНИЯ ДЛЯ TEST COMMANDS

### 5.1. Не принимать работу без фактических команд
Если Codex не выполнил команды проверки и не предоставил фактический вывод, задача не считается завершённой.

### 5.2. Long-running commands не использовать напрямую
В extra_test_commands нельзя добавлять долгоживущие foreground/watch/server-команды без bounded wrapper:
- `python -m uvicorn ...`;
- `npm run dev`;
- `vite`;
- `electron .`;
- аналогичные процессы, которые сами не завершаются.

### 5.3. Ошибки команд нужно классифицировать по фактам
Если команда стартует и падает по assert/error mismatch — это дефект кода/тестов/контракта, а не ограничение среды.
Если команда не может стартовать из-за среды, reviewer должен зафиксировать точную команду, stdout/stderr и причину.

---

## 6. КОРОТКАЯ СУТЬ ДОКУМЕНТА

Этот документ нужен не для описания всей архитектуры проекта, а чтобы:
- не спорить по второму кругу о уже принятых исключениях;
- не путать допустимые отклонения, техдолг, реальные дефекты и ограничения среды;
- удерживать reviewer от лишнего scope;
- помнить реальный техдолг, но чинить его только тогда, когда он действительно входит в текущую задачу;
- принимать результат только по фактическим артефактам Codex: diff, changed files, logs и test summary.
