# PROJECT DOCTRINE — METAPLATFORM

Актуальная консолидированная версия для передачи reviewer в первой итерации и в любых следующих задачах.

Этот документ фиксирует:
- постоянные проектные правила;
- принятые архитектурные решения;
- границы допустимых изменений;
- правила постановки задач reviewer → codex;
- инварианты, которые нельзя размывать между итерациями.

Документ должен оставаться устойчивым к дальнейшим перемещениям репозитория и не должен зависеть от одной конкретной локальной задачи.

---

## 0. БАЗОВОЕ НАПРАВЛЕНИЕ ПРОЕКТА

### 0.1. Что такое MetaPlatform
- MetaPlatform — это модульная инженерная платформа / workbench.
- Это не “обвязка вокруг MetaGen”.
- Платформа должна сохранять самостоятельный core/workbench слой и поддержку нескольких модулей.

### 0.2. Модульность сохраняется
На архитектурном уровне сохраняются модули:
- MetaGen
- MetaLab
- MetaView

Даже если MetaLab и MetaView пока частично реализованы, архитектурно они считаются частью платформы.

### 0.3. Web-only направление зафиксировано
- Electron из целевой архитектуры убирается полностью.
- Целевой UI — web-based IDE.
- Runtime backend существует отдельно от UI.
- Вопросы локального запуска, desktop-shell и packaging desktop app не являются текущим архитектурным приоритетом.

## 1. ЦЕЛЕВАЯ СТРУКТУРА РЕПО

Целевой layout репозитория:

```text
repo/
├─ ui/        # весь frontend
├─ rtb/       # runtime backend
├─ shared/    # shared config / contracts / cross-layer constants
├─ tools/     # helper scripts
├─ docs/      # актуальная документация repo-level
└─ ...        # минимальный root-level набор
```

### 1.1. Правила по layout
- `ui/` содержит frontend целиком: package.json, vite config, index.html, renderer, styles, ui config, public assets при необходимости.
- `rtb/` содержит runtime backend целиком: app, tests, requirements и связанные backend entrypoints.
- `shared/` содержит только действительно shared слой, а не случайный склад файлов.
- `tools/` содержит helper-скрипты проекта.
- `docs/` содержит только актуальные repo-level документы, которые реально должны жить в репозитории.

### 1.2. Что не должно размываться
- Не смешивать в одной задаче repo-layout cleanup и широкие поведенческие рефакторинги без явной необходимости.
- Не держать compatibility-мусор после финализированного переноса, если в задаче явно не требуется временный bridge.
- Не возвращать в helper whitelist и export dump файлы, которые проект уже договорился не экспортировать.

## 2. МОДЕЛЬ ПРОЕКТА

### 2.1. Проект остаётся директорией
Проект = папка.

Внутри проекта архитектурно предполагаются:
- `project.yaml`
- `metagen/`
- `metalab/`
- `metaview/`
- `generated/`

### 2.2. project.yaml
- `project.yaml` обязателен.
- `project.yaml` не хранит полный список документов проекта.
- Список документов определяется сканированием модульных папок.
- `project.yaml` не считается полным source of truth для всей структуры проекта.

### 2.3. Имя проекта
- Имя проекта определяется project file.
- Переименование проекта поддерживается только через Save As.
- Отдельную независимую identity-сущность имени проекта вводить не нужно.

### 2.4. generated/
- `generated/` — это зона результатов генерации.
- Обычный save проекта не должен очищать `generated/`.
- Save проекта и generation outputs — это разные семантики.

---

## 3. IDENTITY И RUNTIME-ПРАВИЛА

### 3.1. Каноническая identity документа
- Единственная каноническая внутренняя identity документа = GUID.
- Внутренние runtime-операции должны строиться только на GUID.

### 3.2. Что НЕ является identity
Не являются identity:
- document name
- file name
- path
- tree label
- tab title

Это display/storage/presentation attributes, но не identity.

### 3.3. Последствия
- rename не должен менять identity документа;
- path change не должен менять identity документа;
- file remap не должен превращаться в смену identity;
- lookup open/active/update/delete/select должен опираться на GUID.

---

## 4. ПРАВИЛА ПО SAVE / PATH / STORAGE

### 4.1. Каноническая save-семантика
Обычный save проекта должен канонически переписывать только project-owned модульные папки:
- `metagen/`
- `metalab/`
- `metaview/`

При этом:
- `generated/` не очищать;
- пользовательские файлы вне project-owned module folders не трогать;
- молчаливое автопереименование для обхода коллизий не считать целевой save-семантикой.

### 4.2. Path semantics
- path-логика должна быть централизована;
- нельзя размазывать одинаковые path rules по UI/runtime/backend слоям;
- пути проекта, project-owned folders, file naming semantics и related helpers должны иметь единый source of truth.

---

## 5. ПРАВИЛА ПО CONFIG / CONSTANTS / MAGIC LITERALS

### 5.1. Общий принцип
Не нужно механически выносить вообще все строки и числа из кода.

Нужно различать:
- допустимые локальные одноразовые literals;
- системные и повторяемые literals, которые являются частью контракта и должны жить в config/constants слое.

### 5.2. Допустимо оставить локально
Допустимо оставлять локально:
- одноразовые error/debug/log messages;
- одноразовые developer-facing тексты;
- одноразовые throw messages;
- локальные сообщения, которые не участвуют в wiring и не повторяются кросс-файлово.

### 5.3. Обязательно выносить
Обязательно выносить в config/constants слой, если значение:
- является id/type/kind/action/reason/status;
- является source key / marker / prefix / suffix с semantic meaning;
- участвует в cross-file wiring;
- повторяется в нескольких местах;
- определяет contract между слоями;
- является timeout / retry / polling / debounce / folder name / extension / endpoint / command id / error code / dialog token / transition token.

### 5.4. Требование к reviewer
Reviewer должен явно различать:
- что вынесено как contract/config/constants;
- что сознательно оставлено локально и почему это допустимо.

## 6. HELPER-СЦЕНАРИЙ ПРОЕКТА

### 6.1. Export helper обязателен
Helper-скрипт проекта:
- `export_project_to_txt.py`

Если в конкретной итерации он позже будет физически перенесён в `tools/`, reviewer должен учитывать целевой layout и синхронизировать пути, wiring и команды под фактическое состояние репозитория.

### 6.2. Export helper работает через ручной whitelist
Это осознанное требование проекта:
- никаких автосканов вместо whitelist по умолчанию;
- список файлов поддерживается вручную;
- whitelist должен оставаться согласованным с фактическим layout репо.

### 6.3. Что export не должен возвращать
В whitelist/export нельзя возвращать:
- иконки;
- demo project / demo-feedmill / аналогичные demo-артефакты;
- top-level tests;
- временные task-documents, которые не должны жить в самом repo;
- `01_project_doctrine.md` и `03_architecture_exceptions.md` как repo artifacts.

Эти два документа могут использоваться как внешние briefing-документы для reviewer, но не должны считаться обязательными файлами самого репозитория.

### 6.4. README / runbook / docs не запрещены автоматически
README, runbook и другая текстовая документация не запрещены по умолчанию.
Если такой файл:
- реально участвует в тестах;
- является входным артефактом для reviewer/codex;
- должен быть синхронизирован с текущим поведением системы;
- явно входит в scope задачи,

то его можно и нужно менять в рамках задачи.

Но это не должно автоматически раздувать задачу до общего docs-cleanup без прямого запроса.

---

## 7. ПРАВИЛА ПО ТЕСТАМ И КОМАНДАМ ПРОВЕРКИ

### 7.1. Полный прогон обязателен
После любых изменений всегда прогоняется полный набор тестов по фактическому проекту.

Нельзя считать задачу завершённой после только таргетных тестов.

### 7.2. Что запрещено в extra_test_commands
Нельзя указывать долгоживущие foreground/watch/server-команды без bounded wrapper, например:
- `python -m uvicorn ...`
- `npm run dev`
- `vite`
- `electron .`
- любые аналогичные процессы, которые сами не завершаются.

### 7.3. Что требуется от reviewer
Reviewer должен в итоговом отчёте перечислять:
- какие тесты были добавлены/обновлены;
- какие команды реально прогнаны;
- что прогнан полный test set;
- что export helper сверен и актуализирован.

---

## 8. ПРАВИЛА ПО ФОРМАТУ ЗАДАЧ ДЛЯ REVIEWER

Каждая задача должна быть:
- небольшой, но законченной;
- с ограниченным scope;
- с явным критерием DONE;
- без расползания на соседние архитектурные темы.

В хорошей задаче reviewer должен получать:
1. что это за отдельная задача и зачем;
2. краткий контекст;
3. границы scope;
4. что сначала проверить;
5. что реализовать;
6. границы свободы;
7. правило по magic literals;
8. требования к тестам;
9. реальные команды проверки;
10. требование сверить export_project_to_txt.py;
11. формат итогового отчёта;
12. критерий DONE.

---

## 9. БЛИЖАЙШИЕ ПРАКТИЧЕСКИЕ ПРИОРИТЕТЫ

Текущий курс repo normalization:
1. `extract-shared-config`
2. `move-ui-into-ui-root`
3. `move-runtime-backend-into-rtb-root`
4. `move-tools-and-docs-and-finalize-repo-layout`

Правило:
- эти шаги нужно делать отдельными небольшими задачами;
- не смешивать весь repo-layout cleanup в один diff;
- после каждого шага синхронизировать imports, команды и whitelist/export.

---

## 10. КОРОТКАЯ ФИНАЛЬНАЯ ФОРМУЛИРОВКА

MetaPlatform развивается как web-only модульная инженерная платформа с целевым repo layout `ui / rtb / shared / tools / docs`.
Проект остаётся директорией с фиксированными модульными папками.
Identity документов строится только на GUID.
Обычный save переписывает только project-owned module folders и не трогает `generated/`.
Path semantics централизуются.
System-level literals выносятся в модульный config/constants слой, а допустимые одноразовые локальные сообщения можно оставлять в коде.
Export helper остаётся whitelist-based и всегда должен соответствовать фактическому layout проекта.
После любых изменений всегда прогоняется полный набор тестов.
