## 0. HARD CONSTRAINTS / НЕ НАРУШАТЬ БЕЗ ОТДЕЛЬНОГО РЕШЕНИЯ

### 0.1. Архитектурное направление не меняем
- MetaPlatform остается контейнером / Workbench, а не “обвязкой вокруг MetaGen”.
- Сохраняется полная архитектура v1:
  - Core
  - Workbench
  - Modules
  - Project
- Сохраняется модульность:
  - MetaGen
  - MetaLab
  - MetaView
- Сохраняется принцип:
  - платформа не должна быть жестко завязана на MetaGen
  - MetaLab и MetaView пока могут быть каркасами, но архитектурно учитываются сразу.

### 0.2. Проект остается директорией
- Проект = папка.
- В проекте есть:
  - project.yaml
  - metagen/
  - metalab/
  - metaview/
  - generated/
- generated/ считается папкой результатов генерации, а НЕ частью обычного канонического save-flow проекта.

### 0.3. project.yaml
- project.yaml обязателен.
- project.yaml НЕ хранит полный список документов проекта.
- Список документов проекта определяется сканированием папок модулей.
- При этом project.yaml НЕ считается полным source of truth для структуры проекта.
- Структура платформы и набор модульных папок фиксированы архитектурой.

### 0.4. Имя проекта
- Имя проекта = имя файла project.yaml / имя project file.
- Переименование проекта поддерживается только через Save As / сохранение как другой проект.
- Поле имени проекта как отдельная независимая identity-сущность не вводится.

### 0.5. Identity документа
- Единственная каноническая identity документа в runtime = GUID.
- Вся внутренняя логика приложения должна строиться только на GUID.
- name документа, file name, path и т.п. не являются identity и не должны использоваться как identity.

### 0.6. generated/ не очищать при обычном save
Это зафиксировано отдельно и окончательно:
- generated/ = результаты генерации
- save проекта != regenerate outputs
- обычный save проекта не должен очищать generated/

### 0.7. Helper-скрипты с ручным whitelist
- Это осознанное требование.
- Не менять.
- `create_project_structure.py` и `export_project_to_txt.py` должны и дальше работать через вручную поддерживаемый список файлов, если это текущий стандарт проекта.

### 0.8. export_project_to_txt.py как whitelist-export, а не “полный честный dump”
- Это тоже осознанное требование.
- Не менять.

### 0.9. IPC / preload / renderer / CJS / ESM boundary
Это критическое архитектурное исключение.

Даже если IPC channel names и IPC handler ids являются системными semantic literals и должны быть централизованы в пределах своего контура, НЕЛЬЗЯ:
- механически объединять preload/main и renderer IPC contract через общий shared config;
- смешивать CJS и ESM boundary ради дедупликации;
- ломать стабильность preload/startup ради “красивой” унификации.

Допустимо:
- осознанное дублирование IPC contract на границе preload / renderer / main,
если это необходимо для стабильности модульной системы и уже является частью архитектуры проекта.

## 1. ОБЩИЕ ПРИНЦИПЫ, КОТОРЫЕ СЧИТАЮТСЯ ЗАФИКСИРОВАННЫМИ

### 1.1. Обязательное правило по тестам
- После любых изменений всегда прогоняется ВЕСЬ набор тестов целиком.
- Не допускается частичный прогон “только рядом затронутых тестов” как финальная проверка.
- Это постоянное правило проекта.

### 1.2. Правило по магическим строкам, числам и config/constants
- Строки и числа внутри корректного config/constants слоя допустимы и считаются нормальным способом хранения конфигурации.
- Не нужно трактовать любые строки и любые числа как автоматически запрещённые.
- Локализацию пока не трогаем и не вводим отдельный i18n-слой.
- Все повторяемые, кросс-файловые, semantic и wiring-related literals должны выноситься в config/constants слой.
- Config/constants слой должен быть модульным по смысловым областям, а не одним giant-config.
- Нельзя оставлять полудубли, когда часть значения уже вынесена в config/constants, а часть рядом всё ещё захардкожена.
- Если literal одноразовый, локальный, не повторяется, не участвует в wiring и не является частью системного контракта, его допустимо оставить локально.
- Если literal является id/type/action/reason/status/channel/handler/marker/prefix/suffix, участвует в wiring, повторяется или определяет cross-file contract, его нельзя оставлять локально.

### 1.3. Что в проекте считается допустимым локальным literal
- одноразовые тексты ошибок;
- одноразовые log/debug/warn messages;
- одноразовые developer-facing сообщения;
- одноразовые throw new Error(...);
- локальные сообщения в commandBus.js, moduleRegistry.js, projectManager.js и других файлах, если они:
  - не повторяются;
  - не участвуют в wiring;
  - не являются логическими токенами;
  - не являются частью кросс-файлового контракта;
  - не используются как ids/types/actions/reasons/channels/handlers/prefixes;
  - не требуют согласования между несколькими файлами.

### 1.4. Что в проекте обязательно выносить в config/constants
- module ids;
- document kinds;
- tree node types;
- tree action ids;
- tab state markers;
- dialog result tokens;
- menu action ids;
- IPC channel names;
- IPC handler ids;
- logger source keys, если они используются как повторяемая классификация;
- transition reason strings;
- prefixes/suffixes/markers, если они несут semantic meaning;
- project folder names;
- file extensions;
- repeated panel ids / panel labels, если они участвуют в wiring;
- repeated CSS class tokens, если JS использует их как логические identifiers;
- default names, если они повторяются или участвуют в contract;
- repeated runtime markers;
- debounce/retry/timeout values;
- repeated layout sizes / min-max / gutter / proportions;
- repeated editor/table behavior constants.

### 1.5. Специальное правило для core-файлов
- В `renderer/core/commandBus.js` допустимо оставлять локально одноразовые error/debug/log strings, но source key, command tokens, dispatch/status/reason tokens и повторяемые semantic markers должны быть вынесены.
- В `renderer/core/moduleRegistry.js` допустимо оставлять локально одноразовые validation/error messages, но registry source key, module ids, document kind tokens, capability/contract markers и повторяемые validation markers должны быть вынесены.
- В `renderer/core/projectManager.js` допустимо оставлять локально одноразовые human-readable error messages и локальные технические пояснения, но project-owned path tokens, folder names, lifecycle/save-state markers, repeated save/rename/remap/status tokens, repeated dialog/save-flow literals и repeated file/path prefixes/suffixes/reasons должны быть вынесены.
- Аналогичное правило применять и к другим файлам проекта:
  - если literal одноразовый, локальный, не повторяется и не является частью контракта — его можно оставить локально;
  - если literal повторяется, участвует в wiring, стал semantic token или является частью архитектурной договорённости — его нельзя оставлять локально.

## 2. ПРИНЯТЫЕ ИСПРАВЛЕНИЯ

### 2.1. Исправить save-семантику по коллизиям имен
Текущее нежелательное поведение:
- система молча разруливает коллизии через автопереименование файлов
- может появляться -1, -2 и т.д.

Принятое решение:
- убрать автоколлизии / автопереименование при обычном save проекта
- save должен работать как каноническая перезапись project-owned содержимого
- при save:
  - очищать metagen/
  - очищать metalab/
  - очищать metaview/
  - затем заново записывать документы из текущего runtime-состояния
- generated/ НЕ очищать при обычном save
- пользовательские файлы вне metagen/, metalab/, metaview/ не трогать
- generated/ остается отдельной зоной результатов генерации

Смысл:
- диск должен быть точной проекцией текущего project runtime по модульным документам
- generated/ не должен страдать от обычного сохранения проекта
- save проекта и generation output должны оставаться разными семантиками

### 2.2. Убрать размазанную и дублирующуюся path-логику
Проблема:
- path-логика и file-path семантика размазаны и дублируются в нескольких местах
- это создает риск расхождения поведения

Принятое решение:
- вынести все правила работы с путями проекта в единый слой / единый модуль
- сделать один source of truth для project path semantics

В этот слой должно входить:
- нормализация путей
- join path
- построение project file path
- определение корня проекта
- определение project-owned директорий
- определение путей модульных документов
- общая семантика “что принадлежит проекту, а что нет”

Правило:
- UI и ProjectManager не должны вручную собирать и дублировать path-логику
- любые операции с путями должны идти через единый utility/service слой

### 2.3. Развязать logger от DOM, сохранив функционал и уровни логов
Проблема:
- logger напрямую завязан на DOM
- это делает core-часть менее чистой и хуже тестируемой

Принятое решение:
- разделить логирование на:
  1) core logger
  2) sinks / adapters

Что должно получиться:
- core logger:
  - знает уровни логов
  - создает log entries
  - фильтрует по level
  - диспатчит записи в sinks
- UI sink:
  - рисует логи в панель Logs / DOM
- console sink:
  - при необходимости зеркалит в console
- memory sink:
  - можно использовать в тестах

Обязательные требования:
- полностью сохранить текущий функционал лог-панели
- полностью сохранить уровни логов
- полностью сохранить понятную работу info / warn / error / debug / none
- сохранить возможность очистки логов
- убрать прямую зависимость core logger от document / DOM API

Правило:
- core пишет в logger
- UI только подписывается на logger и отображает записи

### 2.4. Исправить create_project_structure.py
Проблема:
- скрипт неверно считает количество созданных директорий
- сейчас счетчик увеличивается не по реально созданным директориям, а по более грубой логике

Принятое решение:
- считать только реально созданные директории
- если mkdir(parents=True) создает несколько уровней, учитывать именно реально появившиеся уровни
- не завышать статистику
- корень проекта не считать лишний раз

Ожидаемый результат:
- итоговая статистика скрипта должна отражать реальное количество созданных директорий, а не “уникально встретившиеся parent paths”

## 3. ПРИНЯТЫЕ АРХИТЕКТУРНЫЕ РЕШЕНИЯ ПО СПОРНЫМ ПУНКТАМ

### 3.1. Project Tree: добавить настоящий root-node
Принято:
- в дереве должен быть настоящий корневой узел
- этот узел должен называться НЕ именем проекта, а постоянным именем:
  - Проект

Почему:
- если root-node будет дублировать имя проекта, оно будет повторяться и в header, и в дереве
- это избыточно
- root-node нужен как структурный корень, а не как повтор header

### 3.2. Header панели проекта сохраняется
Принято:
- верхний header над деревом остается
- header выполняет роль status-line

Header должен:
- быть пустым, если проект не открыт
- показывать имя проекта, если проект открыт
- показывать dirty-state / признак несохраненности

То есть:
- header = состояние проекта
- root-node “Проект” = структурный корень дерева

Это два разных уровня UI, не дублирующих друг друга по смыслу.

### 3.3. Стилизовать дерево “как надо”
Принято:
- root-node Проект нужно не просто добавить, а нормально стилизовать все дерево
- визуально различать типы узлов

Нужно сделать:
- отдельный стиль для root-node “Проект”
- отдельный стиль для модулей
- отдельный стиль для документов MetaGen / компонентов
- отдельный стиль для сценариев MetaLab
- отдельный стиль для экранов MetaView

Нужно проверить и исправить:
- сейчас шрифт у модулей меньше, чем у компонентов — привести к осмысленной иерархии
- выделенный узел сейчас становится белым — это неверно
- selected state должен быть темнее, чем hover
- hover и selected должны быть четко различимы
- иерархия должна читаться сразу без всматривания

Допустимо и рекомендуется:
- добавить иконки типов узлов:
  - Проект
  - модуль
  - компонент / документ MetaGen
  - сценарий
  - экран

Важно:
- кнопки добавления / удаления компонентов оставить как есть
- действия на узлах не ломать
- не превращать эту задачу в редизайн всего приложения
- задача именно в том, чтобы дерево стало правильным, читаемым и структурно понятным

### 3.4. GUID как единственная identity
Принято окончательно:
- GUID = единственный канонический идентификатор документа
- все runtime-операции должны работать через GUID

Это означает:
- tree node identity строится по GUID
- tab identity строится по GUID
- active/open/update/delete/select lookup должен опираться на GUID
- path change не должен менять identity документа
- rename display name не должен менять identity документа
- file name не должен менять identity документа

Что НЕ является identity:
- document name
- file name
- path
- label в дереве
- имя вкладки

Они все считаются:
- display attributes
- storage attributes
- presentation attributes
но не identity

### 3.5. Различия document shapes допустимы, но не должны размазываться по общей логике
Принято:
- различия между component / scenario / screen допустимы и нормальны
- но общая логика приложения не должна в разных местах опираться на name/path как на identity

Смысл:
- shape документа может быть разным
- schema payload может быть разной
- но identity-правила едины

Допустимая модель:
- GUID = identity
- module/kind = routing
- name = display
- path/file name = storage

## 5. ВАЖНЫЕ УТОЧНЕНИЯ, ЧТОБЫ НЕ ПРОТИВОРЕЧИТЬ САМИМ СЕБЕ

### 5.1. project.yaml не является полной моделью проекта
Одновременно зафиксировано:
- project.yaml обязателен
- project.yaml не хранит список документов
- project.yaml не задает полностью структуру проекта
- структура проекта определяется архитектурой платформы + сканированием папок

### 5.2. Имя проекта и identity проекта
Зафиксировано:
- имя проекта для UI и статуса определяется именем project file
- переименование проекта = Save As
- не вводить вторую конкурирующую семантику имени

### 5.3. Как трактовать literals внутри config
Зафиксировано:
- строки и числа в config/constants допустимы;
- наличие строк и чисел внутри config/constants не считается недочищенным состоянием само по себе;
- проблема не в literals как таковых, а в literals вне config/constants слоя, если они уже должны быть частью общего контракта.

### 5.4. Что требовать от будущих ТЗ и аудитов по magic literals
Зафиксировано:
- не требовать выноса вообще всех строк и чисел из кода без разбора;
- не считать literals внутри config/constants автоматически проблемой;
- не требовать локализацию как обязательный слой на текущем этапе;
- в commandBus.js, moduleRegistry.js, projectManager.js и аналогичных файлах различать:
  - одноразовые локальные сообщения — допустимо оставить;
  - системные токены и повторяемую семантику — обязательно вынести;
- в итоговом отчёте при зачистке magic literals явно разделять:
  - что вынесено как contract/config/constants;
  - что сознательно оставлено локально и почему.

### 5.5. Как трактовать IPC literals
Зафиксировано одновременно:
- IPC channel names и IPC handler ids являются системными semantic literals;
- их нельзя бездумно оставлять raw в произвольных местах;
- но нельзя и ломать boundary preload / renderer / main ради механической дедупликации;
- любое изменение IPC должно сначала проверять архитектурные ограничения проекта.

   - logger core/sink separation
   - create_project_structure.py directory counting
   - config/constants contracts для вынесенных semantic literals

## 7. ОБЯЗАТЕЛЬНОЕ ПРАВИЛО ДЛЯ ЛЮБОЙ РЕАЛИЗАЦИИ / ТЗ / CODEX-ЗАДАЧИ

Любое дальнейшее ТЗ по этому проекту должно учитывать:
- опора только на актуальный срез проекта
- обязательная проверка helper-скриптов:
  - create_project_structure.py
  - export_project_to_txt.py
- обязательное перечисление конкретных тестов и команд проверки
- обязательный полный прогон всего набора тестов после любых изменений
- запрет на опору на name/path как на identity
- запрет на очистку generated/ при обычном save проекта
- обязательное соблюдение принятой границы для magic strings / magic numbers:
  - config/constants слой — допустимое место для строк и чисел;
  - локальные одноразовые сообщения в коде — допустимы;
  - системные токены, ids, kinds, channels, handlers, actions, reasons, markers и повторяемые semantic literals — обязательно выносить;
  - локализацию пока не вводить и не использовать как обязательное основание для переноса всех пользовательских текстов.
- обязательное соблюдение архитектурного исключения по IPC boundary:
  - не объединять preload/main и renderer contracts через shared config без прямого решения;
  - не смешивать CJS и ESM ради дедупликации;
  - не трогать IPC boundary без необходимости и без проверки архитектурных ограничений.

## 8. КОРОТКАЯ ФИНАЛЬНАЯ ФОРМУЛИРОВКА

MetaPlatform развивается как фиксированная модульная инженерная платформа с проектом в виде директории.
Identity документов строится только на GUID.
Обычный save проекта канонически переписывает только модульные YAML-папки проекта и не трогает generated/.
Project Tree получает настоящий root-node “Проект”, при этом верхний header сохраняется как отдельная status-line.
Path-логика централизуется.
Logger переводится в модель core + sinks.
Подсчет директорий в create_project_structure.py исправляется.
Magic strings / magic numbers приводятся к принятому стандарту: системные токены и повторяемая семантика выносятся в модульный config/constants слой, а допустимые локальные одноразовые literals можно оставлять в коде.
После любых изменений всегда прогоняются все тесты целиком.
IPC boundary между preload / main / renderer не ломается ради формальной дедупликации.
