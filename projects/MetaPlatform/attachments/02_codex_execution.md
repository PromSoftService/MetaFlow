# 02_codex_execution.md

## Назначение документа

Этот документ задаёт стандарт того, **как reviewer должен писать ТЗ для Codex**.

Документ описывает:
- как формулировать следующий implementation step;
- как ограничивать scope;
- как задавать целевую семантику;
- как указывать допустимые файлы для изменений;
- как задавать проверки и запреты;
- как применять правила проекта по magic strings / magic numbers / raw semantic literals;
- как задавать требования к helper-скриптам и проверкам;
- как формулировать требования к тестам и командам.

Этот документ **не** дублирует внешний operational context reviewer input и **не** описывает общий цикл статусов, если он уже задаётся внешними служебными материалами.

---

## 1. БАЗОВАЯ МОДЕЛЬ РАБОТЫ

### 1.1. Разделение ролей
**Reviewer**:
- анализирует текущую задачу и доступные материалы;
- определяет следующий управляемый scope;
- формирует прямое, рабочее ТЗ для Codex;
- после ответа Codex оценивает фактический результат по изменениям и проверкам.

**Codex**:
- работает с реальным репозиторием;
- читает и меняет реальные файлы;
- запускает команды в рабочем дереве репозитория;
- возвращает reviewer фактический результат.

### 1.2. Ключевой принцип
Reviewer анализирует задачу по текущим материалам, но пишет ТЗ так, чтобы оно было **сразу исполнимо Codex внутри репозитория**.

Нельзя писать ТЗ так, как будто Codex редактирует dump, attachment или произвольный reference-файл.

---

## 2. ГЛАВНЫЙ ПРИНЦИП ФОРМУЛИРОВКИ

### Пиши ТЗ напрямую для Codex
Каждый task должен быть написан так, как будто ты обращаешься напрямую к Codex, который:
- уже находится в репозитории;
- может читать реальные файлы;
- может запускать команды;
- может вносить изменения;
- может сам локализовать нужные места по коду.

Нельзя писать:
- reviewer-заметки вместо ТЗ;
- abstract guidance без следующего шага;
- summary проверки без прямых инструкций;
- полунамёки и неопределённые пожелания.

Нужно писать:
- какой именно текущий scope;
- что сначала проверить;
- какую целевую семантику нужно получить;
- что конкретно изменить;
- какие файлы можно менять;
- что запрещено делать;
- какие проверки обязательно выполнить;
- в каком формате отчитаться.

---

## 3. ЧТО ТАКОЕ ХОРОШЕЕ ТЗ ДЛЯ CODEX

Хорошее ТЗ — это текст, после которого Codex понимает:

1. какой именно текущий scope итерации;
2. что именно сначала нужно проверить в репозитории;
3. какую целевую семантику нужно получить;
4. что конкретно нужно изменить;
5. какие файлы можно менять;
6. что запрещено делать;
7. какие проверки обязательно выполнить;
8. в каком формате отчитаться.

Если после чтения ТЗ Codex может:
- пойти в соседний scope;
- сделать лишний рефакторинг;
- пропустить важную проверку;
- оставить approximation вместо точного исправления;
- выполнить только verification-only pass без конкретной доработки,

значит ТЗ написано плохо.

---

## 4. ОБЯЗАТЕЛЬНЫЙ СТИЛЬ

### 4.1. Пиши операционно, а не рассуждательно
Плохой стиль:
- “возможно, стоит”;
- “кажется, лучше”;
- “можно попробовать”;
- “наверное, проблема в …”.

Хороший стиль:
- “сначала проверь …”;
- “найди …”;
- “сопоставь …”;
- “исправь …”;
- “не меняй …”;
- “запусти …”;
- “перечисли …”.

### 4.2. Пиши короткими инженерными инструкциями
Каждый блок должен отвечать на конкретный вопрос:
- что проверить;
- что изменить;
- чего не делать;
- как проверить результат.

### 4.3. Пиши прямо для исполнителя
Используй прямую форму:
- “проверь”;
- “найди”;
- “исправь”;
- “обнови”;
- “не трогай”;
- “запусти”;
- “проверь результат”.

---

## 5. КАНОНИЧЕСКАЯ СТРУКТУРА TASK

Используй эту структуру по умолчанию.

```md
# Context
Краткий контекст текущей итерации и текущего scope.

# Current scope
Один управляемый шаг без расползания на соседние темы.

# Target semantics
Что именно должно быть истинно после правки.

# What to inspect first
Какие файлы, функции, сценарии, слои и связи сначала проверить.

# Required changes
Нумерованный список конкретных изменений.

# Files allowed to change
Точные файлы или допустимая область изменений.

# Do not do
Явные запреты для текущего scope.

# Verification
Конкретные команды и проверки после изменений.

# Result report
Требование к структурированному итоговому отчёту.
```

---

## 6. КАК ВЫБИРАТЬ ТЕКУЩИЙ SCOPE

### 6.1. Один task = один управляемый scope
Не смешивай в один task:
- архитектурный рефакторинг;
- новую функциональность;
- широкий cleanup;
- тестовую инфраструктуру;
- визуальную доводку;
- миграцию naming.

Если это разные темы, разделяй.

### 6.2. Scope должен быть достаточно узким
Хороший scope:
- конкретный;
- завершимый;
- проверяемый;
- связан с текущим расхождением или следующим логическим шагом.

Плохой scope:
- “приведи проект в порядок”;
- “доведи модуль до идеала”;
- “полностью синхронизируй всё поведение”.

### 6.3. Не переходи к следующему scope раньше времени
Если текущий scope ещё:
- approximation;
- с неполной семантикой;
- с незакрытым cleanup;
- с заметными расхождениями;
- с потенциальными побочными эффектами,

нужно писать task на доработку текущего scope, а не переходить к новой теме.

---

## 7. КАК ФОРМУЛИРОВАТЬ TARGET SEMANTICS

Reviewer должен не просто сказать “что сломано”, а объяснить Codex, **какая целевая семантика должна быть получена**.

Плохо:
- “исправь сохранение”;
- “почини рендер”;
- “убери баг”.

Хорошо:
- “после правки сохранение должно сбрасывать dirty-state только после успешного полного project-level save”;
- “после правки вычисление должно использовать один источник истины для X”;
- “после правки компонент должен сохранять текущий внешний контракт и не менять поведение соседних сценариев”.

Если целевая семантика не сформулирована, Codex слишком легко уходит в формально работающую, но неверную реализацию.

---

## 8. КАК ПИСАТЬ WHAT TO INSPECT FIRST

Этот блок обязателен.

Reviewer должен сначала направить Codex:
- где искать текущую реализацию;
- какие соседние места проверить;
- какие точки интеграции нельзя пропустить.

Нужно указывать:
- конкретные пути, если они известны;
- названия модулей, функций, компонентов, обработчиков, сервисов;
- связи между слоями;
- likely source of truth;
- likely duplicated logic;
- места возможных side effects.

---

## 9. КАК ПИСАТЬ REQUIRED CHANGES

Это главный блок.

### 9.1. Пиши нумерованным списком
Не сплошным абзацем, а проверяемыми пунктами.

### 9.2. Каждый пункт должен быть предметным
Плохой пункт:
- “приведи код в порядок”.

Хороший пункт:
- “убери дублирование вычисления X между A и B, оставив один источник истины в C”;
- “обнови вызов D так, чтобы он использовал уже вычисленное значение E”;
- “сохрани внешний контракт метода F без изменения вызывающих мест”.

### 9.3. Разделяй “проверить” и “изменить”
Не смешивай диагностику и правку в одном туманном предложении.

### 9.4. Избегай расплывчатых слов
Не пиши:
- “подчисти”;
- “гармонизируй”;
- “сделай красиво”;
- “сделай правильно”.

---

## 10. FILES ALLOWED TO CHANGE

Этот блок нужен, чтобы ограничить расползание Codex.

### 10.1. Если знаешь точные файлы — перечисли их
Это лучший вариант.

### 10.2. Если точные файлы не знаешь — перечисли область
Например:
- только `ui/renderer/runtime/**`;
- только `rtb/app/services/**`;
- только helper-слой и связанные docs.

### 10.3. Явно укажи отношение к тестам
Нужно писать:
- можно менять тесты или нет;
- в каком объёме;
- только адаптация к корректной семантике или можно добавлять новые tests.

### 10.4. Явно укажи отношение к helper/config/docs
Отдельно укажи, можно ли менять:
- helper-скрипты;
- config/constants;
- README/runbook/docs;
- build/package files.

Если scope затрагивает helper/export/repo consistency, reviewer должен явно разрешать изменения в соответствующих helper-файлах.

---

## 11. DO NOT DO

Это обязательный блок.

Типовые запреты:
- не меняй несвязанные участки проекта;
- не делай большой рефакторинг без прямой необходимости;
- не меняй публичный API, если задача не про это;
- не меняй layout / naming / structure за пределами текущего scope;
- не заменяй точную логику приближением;
- не редактируй dump;
- не генерируй новый dump, если это отдельно не разрешено;
- не маскируй проблему отключением проверки;
- не подгоняй тесты под неправильную реализацию;
- не оставляй временные заглушки;
- не вводи новые magic strings / magic numbers / raw semantic literals без необходимости и без существующего constants/config слоя.

Если существует конкретный нежелательный путь, который Codex может выбрать, запрети его явно.

---

## 12. ПРАВИЛА ПО CONFIG / CONSTANTS / MAGIC LITERALS

### 12.1. Общий принцип
Reviewer обязан требовать от Codex не просто “не плодить магические литералы”, а соблюдать следующую норму:

- семантические литералы должны выноситься в существующий config/constants слой;
- локальные технические литералы допустимо оставлять в коде, если они не образуют общий контракт и не создают новый source of truth;
- при сомнении литерал нужно трактовать как кандидата на вынос.

### 12.2. Что считать обязательным кандидатом на вынос
Выносить обязательно:
- module ids;
- document kinds;
- node types;
- action ids;
- command ids;
- status-like values;
- reason ids;
- outcome ids;
- source ids;
- endpoint paths;
- timeout / retry / polling values;
- folder names / file names / path prefixes;
- shared dialog tokens;
- устойчивые UI labels, если они являются частью контракта;
- shared policy/config values.

### 12.3. Что можно оставлять локально
Обычно можно оставлять локально:
- `0`, `1`, `-1`, `index + 1` и подобные простые алгоритмические значения;
- одноразовые internal guard strings;
- platform API literals;
- маленькие технические literals внутри узкого helper-а, если они не повторяются и не образуют contract surface.

### 12.4. Что reviewer должен писать в task
Если scope затрагивает магические литералы, reviewer должен писать это явно, например:
- “Не оставляй новые raw semantic literals в рабочем коде.”
- “Всё, что является shared semantic/config value, вынеси в существующий config/constants слой.”
- “Не выноси механически локальные `0/1/index/offset` literals, если они не образуют shared contract.”
- “Не создавай новый параллельный слой констант, если уже есть подходящий config/constants слой.”

### 12.5. Что reviewer должен проверять после изменений
В блоке Verification reviewer должен отдельно требовать проверить:
- не появились ли новые raw semantic literals;
- не остались ли дублирующиеся semantic values в коде;
- не возник ли второй competing source of truth;
- не случился ли вредный over-extraction микролитералов.

## 13. HELPER-СКРИПТЫ И EXPORT HELPER

### 13.1. Export helper обязателен
Helper-скрипт проекта:
- `export_project_to_txt.py`

Если scope затрагивает helper/export/repo consistency, reviewer должен прямо разрешать и требовать правки этого файла.

### 13.2. Export helper работает через ручной whitelist
Reviewer должен требовать:
- никаких автосканов вместо whitelist по умолчанию;
- список файлов поддерживается вручную;
- whitelist должен оставаться согласованным с фактическим layout репо и текущим рабочим кодом.

### 13.3. Что export нельзя включать
При задачах, затрагивающих export helper, reviewer должен отдельно требовать, чтобы в export не попадали:
- иконки;
- тесты;
- demo artifacts;
- briefing docs;
- любые package-lock.json;
- временные task-documents, которые не должны жить в repo.

### 13.4. Что export может включать
Whitelist не должен искусственно ограничиваться заранее зафиксированным набором директорий.
Если в репо появились новые нужные рабочие файлы, они могут быть добавлены в whitelist, если это соответствует scope и не нарушает явные запреты.

---

## 14. ПРАВИЛА ПО ТЕСТАМ И КОМАНДАМ ПРОВЕРКИ

### 14.1. Полный прогон обязателен
После любых изменений reviewer должен требовать полный набор тестов по фактическому проекту, а не только таргетные проверки.

Для текущего проекта Codex должен прогонять все test scripts из `ui/package.json`, относящиеся к тестам:
- `npm.cmd test`
- `npm.cmd run test:integration`
- `npm.cmd run test:backend`
- `npm.cmd run test:all`

Reviewer должен отдельно проверить, что Codex реально прогнал эти команды и привёл фактические результаты. :contentReference[oaicite:0]{index=0}

### 14.2. Команды должны соответствовать фактическому layout
Reviewer должен сверять test/run commands, helper expectations и docs с текущим состоянием репозитория.

Если scripts, пути тестов, backend test roots, helper-ожидания или README/run instructions расходятся с фактическим layout репо, reviewer должен требовать их синхронизации в рамках задачи, если это входит в scope. :contentReference[oaicite:1]{index=1}

### 14.X. Диагностика failing startup probe и других bounded test-сценариев

Если нужно диагностировать failing startup probe или другой узкий backend test failure, reviewer может разрешать Codex запускать только сам pytest-тест в bounded-форме, например через `python -m pytest ...::test_name ...`, если это обычная завершаемая test command.

Reviewer должен запрещать Codex запускать напрямую долгоживущие foreground/server/watch-команды, включая:
- `python -m uvicorn ...`
- `npm run dev`
- `vite`
- `electron .`
- любые аналогичные процессы, которые сами не завершаются.

Если для диагностики действительно нужен прямой запуск server-процесса, reviewer может разрешать это только через bounded wrapper:
- с явным timeout;
- с гарантированным завершением процесса;
- с обязательной фиксацией stdout/stderr;
- с явным объяснением, почему одного pytest-запуска недостаточно.

Нельзя использовать прямой server run как дефолтный путь проверки, если достаточно bounded pytest-команды.

### 14.4. Явные команды
Не писать “прогони тесты”.

Reviewer должен явно указывать:
- какие именно команды запустить;
- в каком порядке;
- что полный прогон обязателен;
- нужны ли дополнительные targeted команды сверх полного прогона.

Для текущего проекта базовый обязательный набор:
- `npm.cmd test`
- `npm.cmd run test:integration`
- `npm.cmd run test:backend`
- `npm.cmd run test:all` :contentReference[oaicite:2]{index=2}

### 14.5. Обязательный подход к backend pytest temp directories
Для backend pytest reviewer должен **по умолчанию требовать использовать уже существующий project-level подход из `conftest.py`**, который перенаправляет `tmp_path` в отдельный writable temp root.

Этот механизм должен использоваться как стандартный путь запуска backend tests, а не как разовый workaround “если вдруг упадёт”.

Reviewer должен требовать от Codex:
- использовать существующий `conftest.py`-подход для `tmp_path`;
- не возвращаться к стандартному `pytest tmp_path` поведению поверх проблемного ACL/path окружения;
- не придумывать локальные ad-hoc обходы в отдельных тестах;
- если backend tests всё равно падают, приложить точный `stderr/stdout` и явно указать, что проблема сохраняется **при уже используемом project-level temp-dir approach**. :contentReference[oaicite:3]{index=3}

### 14.6 Codex не использовать команду `pytest` напрямую
Reviewer должен поручать Codex запускать pytest только в форме `python -m pytest ...`.
Если тестам не нужен pytest cache, reviewer должен по умолчанию добавлять `-p no:cacheprovider`, чтобы Codex не пытался писать `.pytest_cache` внутри repo.

### 14.7. Проверка helper-скриптов
Если helper-скрипты входят в scope, reviewer должен прямо указывать:
- проверять их только статически или запуском;
- допустима ли генерация dump;
- нужно ли актуализировать whitelist и пути;
- какие файлы явно запрещено включать.

---

## 15. VERIFICATION

Этот блок должен быть жёстким и конкретным.

### 15.1. Проверка на побочные эффекты
Отдельно требуй:
- проверить соседние сценарии;
- проверить, что не сломан контракт;
- проверить, что изменение не породило вторую конфликтующую ветку поведения.

### 15.2. Проверка на технические хвосты
Требуй:
- не оставлять временные ветки;
- не оставлять мёртвый код;
- не оставлять partially-updated naming;
- не оставлять несинхронизированные helper/docs/config fragments.

---

## 16. RESULT REPORT

Требуй короткий, но структурированный отчёт.

Минимум:
- изменённые файлы;
- что изменено по каждому файлу;
- какие тесты и команды запущены;
- результаты команд;
- есть ли риски;
- есть ли ограничения;
- остались ли открытые вопросы.

Хороший формат:
- Summary;
- Changed files;
- What changed;
- Verification;
- Risks / open items.

---

## 17. WINDOWS EXECUTION SPECIFICS

### 17.1. npm on PowerShell
Для Windows/PowerShell reviewer должен ориентировать Codex использовать `npm.cmd`, а не `npm`, чтобы не упираться в execution policy для `npm.ps1`.

Примеры:
- `npm.cmd test`
- `npm.cmd run test:integration`
- `npm.cmd run test:backend`
- `npm.cmd run test:all`

### 17.2. Backend tests and local temp directory
Если backend-тесты падают из-за permission issues в системном temp-каталоге, reviewer не должен сразу трактовать это как дефект продуктового кода.

Сначала нужно:
- подтвердить проблему по stderr/stdout;
- отделить environment issue от реального дефекта;
- при необходимости направить Codex на bounded запуск с локальным temp-каталогом внутри репозитория.

### 17.3. Validation principle
Если команда не стартовала из-за shell/policy/temp/cache/environment issue, это не считается доказательством дефекта продуктового кода.

Reviewer обязан различать:
1. проблемы запуска команды;
2. ограничения execution environment;
3. реальные падения тестов по логике проекта.

## 18. КОРОТКАЯ ФОРМУЛА

Reviewer анализирует текущие материалы и формирует прямое ТЗ для Codex.
Codex меняет реальные файлы репозитория и запускает реальные команды.
Поэтому reviewer обязан писать компактные, предметные, операционные tasks с точным scope, явной target semantics, явными запретами, конкретной verification и постоянным контролем magic strings / magic numbers по правилам проекта.
