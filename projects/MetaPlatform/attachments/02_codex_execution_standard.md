# TECH SPEC — Codex Execution Standard (MetaPlatform)

## 1. Общие принципы

Работай **только от текущего состояния репозитория** (актуальный срез в папке `repo`).

НЕ опирайся на:
- прошлые версии проекта,
- предположения,
- внешние знания, если они не подтверждаются кодом.

Каждое решение должно быть основано на:
- текущем коде,
- текущей структуре проекта,
- фактических зависимостях.

---

## 2. Главная цель задачи

Выполнить задачу **полностью и консистентно**:

- не оставлять частичных решений,
- не оставлять “полумодель”,
- не создавать временных/локальных костылей.

Любое изменение должно:
- быть согласовано со всей системой,
- не ломать существующую архитектуру,
- не создавать скрытых расхождений.

---

## 3. Работа с магическими строками и числами

### Обязательное правило

Если значение является:
- semantic token,
- частью контракта,
- идентификатором (IPC, module, type, action и т.д.),
- или используется более одного раза,

→ оно должно быть вынесено в config/constants слой.

### НЕ выносить

- локальные значения внутри одной функции,
- test fixtures, не являющиеся частью контракта,
- одноразовые значения без повторного использования.

### Обязательная консистентность

Если значение вынесено в config:

- оно должно использоваться **везде**,
- не допускается дублирование старого литерала рядом с config,
- не допускается “частичное внедрение”.

---

## 4. Критическое архитектурное исключение (ВАЖНО)

### IPC / preload / renderer / CJS / ESM boundary

НЕ допускается:

- объединение IPC контрактов между:
  - preload (CJS),
  - renderer (ESM),
  - main,

через общий shared config,

ЕСЛИ это нарушает границы модульной системы.

### Допустимо

- осознанное дублирование IPC контрактов,
- если это необходимо для стабильности boundary,
- и если это уже часть архитектуры проекта.

### Перед изменениями

Перед любыми изменениями в:
- IPC,
- preload,
- config,
- module system,

обязательно:

1. проверить существующие ограничения в коде,
2. учитывать архитектурные исключения,
3. не применять “универсализацию” без проверки.

---

## 5. Проверка helper-скриптов (обязательно)

Всегда проверить:

- `create_project_structure.py`
- `export_project_to_txt.py`

Проверить:
- не сломаны ли,
- не требуют ли обновления,
- корректно ли отражают новую структуру проекта.

---

## 6. Тестирование (обязательно)

После любых изменений:

1. Выполнить **полный набор тестов**.
2. Использовать команды, предусмотренные проектом.
3. Не ограничиваться только точечными тестами, даже если менялась малая часть кода.

Если менялись:
- preload,
- IPC,
- config,
- startup path,
- module loading,
- Electron boundary,

то дополнительно обязательно проверить, что приложение **запускается без runtime crash**.

---

## 7. Повторная перепроверка после тестов

После завершения изменений и полного прогона тестов необходимо повторно проверить весь репозиторий и убедиться, что:

- не осталось частично вынесенных литералов,
- не осталось старых дублирующих значений рядом с новыми constants/config,
- не возникло новых архитектурных расхождений,
- helper-скрипты соответствуют новой структуре,
- изменения завершены полностью.

---

## 8. Что нельзя делать

НЕ допускается:

- оставлять частично выполненную зачистку,
- выносить часть токенов и оставлять рядом старые raw literals того же типа,
- централизовать всё подряд без учета архитектурных исключений,
- менять IPC boundary без крайней необходимости,
- смешивать CJS и ESM контракты ради “красоты”,
- считать задачу выполненной без полного прогона тестов,
- считать задачу выполненной без повторной финальной проверки репозитория.

---

## 9. Формат итогового отчета Codex

Итоговый отчет должен быть **кратким, структурированным и без лишнего сырого вывода**.

Нужно указать:

### 1. Summary
Кратко:
- что сделано,
- зачем сделано,
- завершена ли задача полностью.

### 2. Changed files
Список реально измененных файлов.

### 3. What changed
Кратко по сути изменений:
- какие контракты/литералы/слои были изменены,
- какие места были дочищены,
- какие helper-скрипты были обновлены.

### 4. Tests summary
Только краткий итог по тестам:
- какие команды были выполнены,
- статус каждой команды: `passed` / `failed`,
- если есть ошибка — только релевантный фрагмент.

**Не вставлять полный stdout успешных тестов.**

### 5. Risks / Notes
Указать:
- трогались ли high-risk зоны,
- были ли архитектурные ограничения,
- что сознательно НЕ менялось.

---

## 10. Каналы данных для reviewer

### Технический канал
Канонический технический канал должен включать:

- `git diff`
- `git diff --stat`
- `changed_files`
- `tests_summary.json`

### Модельный канал
Канонический модельный канал должен включать только финальный structured summary от Codex.

Не нужно передавать reviewer полный сырой stdout/stderr Codex, если это не требуется для debug/escalation сценария.

---

## 11. Debug / escalation режим

Полный сырой вывод, расширенные логи и дополнительные подтверждения нужны только если:

- тесты упали,
- есть runtime crash,
- есть спорная ситуация,
- пользователь отдельно попросил доказательства,
- reviewer не может принять решение по компактным артефактам.

Во всех остальных случаях использовать компактный отчет.

---

## 12. Финальный критерий готовности

Задача считается завершенной только если одновременно выполнено всё:

- изменения внесены полностью,
- не осталось недочищенных магических строк/чисел того же класса,
- архитектурные исключения не нарушены,
- helper-скрипты проверены,
- полный набор тестов прогнан,
- runtime не сломан,
- репозиторий повторно перепроверен после тестов,
- итоговый отчет отражает реальное состояние проекта.
