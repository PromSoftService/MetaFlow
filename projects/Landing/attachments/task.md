# 3. ГЛАВНОЕ ПРАВИЛО РАБОТЫ

Сначала привести к нужному виду то, что уже реализовано.
Только потом двигаться дальше.

Ты обязан работать строго в таком режиме:

1. Inspect existing implementation:
   - сравнить уже существующую реализацию в `index.html` + `styles.css` с `landing.html`.

2. Refine existing implementation:
   - если уже реализованная часть ещё заметно далека от `landing.html`, сначала дать Codex task на её приближение.

3. Add exactly one new block:
   - только если уже реализованная часть стала достаточно близкой к `landing.html`, разрешается добавить ровно один следующий новый блок.

4. Verify that new block:
   - после добавления нового блока снова проверить результат против `landing.html`.

5. Refine that block until close enough:
   - если новый блок ещё заметно расходится с исходником, сначала довести его.

6. Only then move further:
   - только потом можно разрешать следующий новый блок.

Иными словами:
- inspect what already exists;
- refine what already exists;
- add exactly one new block;
- verify it;
- refine it;
- only then move further.

# 10. АНТИ-МИКРООПТИМИЗАЦИОННОЕ ПРАВИЛО

Ты не имеешь права застревать в микрооптимизации уже реализованного scope.

Если уже реализованная часть:
- визуально стабильна;
- не содержит грубых композиционных или структурных расхождений;
- не имеет явных регрессий;
- а оставшиеся отличия вторичны, локальны и не дают заметного пользовательского эффекта без прямого попиксельного сравнения,

то ты обязан считать этот scope достаточно близким и двигаться к ровно одному следующему новому блоку.

Запрещено удерживать процесс на одном и том же scope ради:
- малозаметных микро-сдвигов spacing;
- косметических поправок без заметного визуального эффекта;
- повторных verification-only pass;
- ещё одной итерации “на всякий случай”, если уже нет явных грубых расхождений.

Если две подряд итерации в одном и том же scope не дали заметного визуального прогресса или дали только микро-правки без ощутимого эффекта, reviewer обязан:
- прекратить refinement этого scope;
- признать его accepted baseline;
- перевести Codex на ровно один следующий новый блок.

Исключение:
если в уже реализованном scope есть явный функциональный дефект, заметная регрессия, сломанная адаптивность, неверные ссылки/якоря/контакты или грубое визуальное расхождение с `landing.html`, тогда refinement нужно продолжать до устранения именно этих проблем.

# 12. ФОРМАТ ТВОЕГО ОТВЕТА

В ответе должны быть:

## `summary`
Кратко:
- existing implementation inspected;
- existing part is / is not close enough;
- therefore next step is refine existing implementation OR add exactly one new block.

## `codex_task_md`
Точный task для Codex:
- либо на добивку уже реализованной части;
- либо на добавление ровно одного следующего нового блока.

## `review_notes`
Ты обязан явно написать одно из двух:
- `existing implementation is not yet close enough; continue refinement before adding new block`
- `existing implementation is close enough; add exactly one next block`

И кратко указать основания.

Если ты оставляешь текущий scope на refinement, ты обязан назвать конкретные заметные проблемы, которые ещё требуют исправления.
Если заметных проблем уже нет, ты не имеешь права оставлять scope на ещё одну микроитерацию.

## `extra_test_commands`
Только те команды, которые реально помогают проверить текущий scope.

Критическое правило:
сначала привести к нужному виду то, что уже есть в `index.html` и `styles.css`;
только потом двигаться дальше;
дальше всегда идти по схеме:
- add one new block;
- verify it;
- refine it to close match;
- only then add the next block.

Для первой итерации запрещено добавлять новый блок.
Первая итерация используется только для:
1. audit уже реализованной части;
2. исправления расхождений в уже реализованной части, если они есть;
3. фиксации, достаточно ли близка уже реализованная часть к source of truth.

Решение о добавлении следующего нового блока допускается только в следующей reviewer-итерации, после получения и проверки diff по исправлениям уже существующей реализации.
